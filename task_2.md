# Задание 2. Электронно-цифровая подпись

Во многих автоматизированных системах необходимо гарантировать, что:
- сообщение действительно отправлено конкретным субъектом;
- содержимое сообщения не было изменено при передаче;
- отправитель не может отказаться от факта отправки сообщения.

Для решения этих задач используется **электронно-цифровая подпись (ЭЦП)**, основанная на криптосистемах с открытым ключом.

---

## Общий принцип работы ЭЦП

![/data/eds.png](/data/eds.png)
> *Рис. 3 — Общий принцип работы электронной цифровой подписи*

Механизм ЭЦП базируется на использовании **асимметричного шифрования**, при котором каждому участнику соответствует пара ключей:

- **открытый ключ (public key)** — доступен другим участникам;
- **закрытый ключ (private key)** — хранится только у владельца.

Ключи связаны между собой математически таким образом, что:
- сообщение, зашифрованное закрытым ключом, может быть проверено только с помощью соответствующего открытого ключа;
- восстановить закрытый ключ по открытому за разумное время невозможно.

Открытый ключ обычно распространяется в составе сертификата, а закрытый ключ никогда не покидает доверенную среду владельца.

---

## Подпись сообщения

Пусть существуют два абонента — **A** и **B**.

Общая схема подписания и проверки сообщения выглядит следующим образом:

1. Абонент **A** вычисляет хэш от исходного сообщения.
2. Полученный хэш шифруется **закрытым ключом A** — таким образом формируется подпись.
3. Абонент **A** передаёт сообщение вместе с подписью абоненту **B**.
4. Абонент **B**, используя **открытый ключ A**, расшифровывает подпись и получает хэш.
5. Абонент **B** самостоятельно вычисляет хэш от полученного сообщения и сравнивает его с расшифрованным.

Если хэши совпадают, сообщение считается подлинным.

---

## Какие задачи решает ЭЦП

Использование электронной подписи позволяет защититься от следующих нарушений:

- отправитель не может отказаться от авторства сообщения, если закрытый ключ известен только ему;
- злоумышленник не может подделать подпись без знания закрытого ключа;
- изменение сообщения приводит к нарушению целостности и обнаруживается при проверке.

При необходимости схема может быть дополнена **шифрованием содержимого сообщения**, однако это зависит от области применения и не является обязательным для самой подписи.

## Задание

Необходимо разработать программное решение, демонстрирующее практическое применение электронной цифровой подписи в клиент–серверном взаимодействии.

Реализация должна поддерживать **два сценария**, отражающие разные варианты использования ЭЦП.

---

### Сценарий 1. Подпись на стороне клиента

В данном сценарии клиент самостоятельно формирует и подписывает сообщение.

Последовательность действий:

1. Клиент формирует произвольное сообщение.
2. Клиент подписывает сообщение с использованием собственного закрытого ключа.
3. Клиент отправляет подписанное сообщение на сервер.
4. Сервер выполняет верификацию подписи, используя публичный ключ клиента.
5. Сервер возвращает клиенту результат проверки.
6. Клиент отображает полученный статус верификации.

---

### Сценарий 2. Подпись на стороне сервера

В данном сценарии сервер выступает источником подписанного сообщения.

Последовательность действий:

1. Клиент запрашивает у сервера публичный ключ.
2. Клиент инициирует запрос на генерацию случайного сообщения.
3. Сервер формирует сообщение и подписывает его своим закрытым ключом.
4. Сервер передаёт клиенту сообщение и подпись.
5. Клиент выполняет верификацию подписи, используя публичный ключ сервера.
6. Клиент определяет, является ли сообщение подлинным.
